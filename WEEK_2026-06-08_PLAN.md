# Личный рабочий план по PyBot ITAcadem на 8-12 июня 2026

## Назначение недели

Эта неделя нужна не для рывка и не для закрытия всего июньского backlog. Её задача — вернуть работе конечность и подготовить спокойный переход к PostgreSQL, не начиная одновременно MCP, новые продуктовые функции и очередной большой инфраструктурный эпик.

## Главный результат

К концу недели должно быть понятно и проверяемо:

- в каком состоянии находится текущий release/deploy-контур после последних изменений;
- как именно проект будет переходить с SQLite на PostgreSQL;
- какие реальные несовместимости нужно исправить;
- какой следующий небольшой implementation slice можно брать без повторного исследования всей системы.

Полностью завершать миграцию на PostgreSQL на этой неделе не требуется.

## Минимально успешная неделя

Неделя считается состоявшейся, если:

- текущая ветка и PR приведены в понятное состояние;
- выполнены targeted checks последних release/deploy изменений;
- принято и записано решение о стратегии PostgreSQL-миграции;
- составлен проверенный список затрагиваемых моделей, миграций, runtime и deploy-файлов;
- создан черновик нового ADR, superseding ADR-0004.

Если сделано только это, неделя не считается проваленной.

## Хороший дополнительный результат

Только при нормальном самочувствии и без форсирования:

- чистая тестовая PostgreSQL принимает текущую схему через Alembic;
- подтверждён список dialect-specific проблем;
- исправлена exit semantics seed-процесса и добавлены targeted tests.

## Не делать на этой неделе

- не начинать реализацию MCP-сервера;
- не добавлять MCP write tools, JWT или SQLAdmin;
- не начинать систему задач;
- не восстанавливать monitoring-домен параллельно с PostgreSQL;
- не проводить ещё один полный code/security audit;
- не рефакторить весь слой текстов;
- не добавлять Webhook, FastStreams или новые Messaging Patterns;
- не заниматься необязательным polishing после прохождения нужной проверки;
- не принимать новые задачи без снятия одной из задач этой недели.

## Режим работы

- Одна основная задача в работе.
- Нормальный день: один основной блок на 60-90 минут и один лёгкий блок на 30-60 минут.
- Минимальный день: один блок на 25-45 минут с фиксацией результата.
- После каждого блока записывать следующую конкретную точку входа в работу.
- Не компенсировать пропущенный день удвоенной нагрузкой на следующий.
- При физическом ухудшении, сильной тревоге или невозможности сосредоточиться работа по проекту прекращается без отработки долга вечером.
- Не планировать проект на выходные заранее. Выходные остаются резервом отдыха, а не скрытым дедлайном.

## Понедельник, 8 июня

### Фокус: закрыть текущий контекст

- Просмотреть состояние текущего PR и незакоммиченных документов.
- Кратко зафиксировать, что именно изменилось в release/deploy/runtime-контуре.
- Запустить только targeted tests для:
  - deploy environment validation;
  - release history validation;
  - Compose runtime contract;
  - DI/runtime, если он был затронут.
- Записать найденные ошибки отдельно, не переключаясь сразу на PostgreSQL.

**Достаточный результат дня:** текущая работа имеет понятный статус, а следующий шаг не нужно восстанавливать по истории Git.

## Вторник, 9 июня

### Фокус: определить контракт миграции

- Перечитать ADR-0004 и отметить решения, которые больше не соответствуют текущему runtime.
- Выбрать стратегию:
  - clean start на PostgreSQL; или
  - перенос существующих SQLite-данных.
- Определить, требуется ли поддержка SQLite после перехода. Предпочтительно не поддерживать две production СУБД.
- Начать ADR о возвращении на PostgreSQL:
  - контекст;
  - причины;
  - решение;
  - последствия;
  - rollout и recovery.

**Достаточный результат дня:** принято одно миграционное решение, даже если код ещё не менялся.

## Среда, 10 июня

### Фокус: ограниченный технический аудит PostgreSQL

- Проверить только относящиеся к миграции участки:
  - PK/FK и смешение `Integer`/`BigInteger`;
  - task-модели как deferred schema;
  - enum/string fields;
  - `ondelete`, unique constraints и индексы;
  - Alembic `env.py`;
  - seed/bootstrap;
  - health probe и database settings.
- Составить список:
  - точно требует изменения;
  - вероятно совместимо;
  - нужно подтвердить экспериментом.
- Не исправлять всё по ходу чтения.

**Достаточный результат дня:** миграция разбита на конечные технические slices.

## Четверг, 11 июня

### Фокус: первый эксперимент

- Поднять отдельную чистую PostgreSQL для проверки, не затрагивая текущие данные.
- Попробовать применить текущий Alembic history к чистой базе.
- Зафиксировать фактические ошибки и предупреждения.
- Если schema upgrade проходит, выполнить минимальную проверку подключения и repository smoke.
- Не начинать массовые правки моделей без результата эксперимента.

**Достаточный результат дня:** известен фактический, а не предполагаемый объём несовместимости.

## Пятница, 12 июня

### Фокус: завершить неделю, а не открыть новый эпик

- Свести результаты проверки PostgreSQL в короткий migration checklist.
- Дополнить ADR фактическими выводами.
- Определить первый implementation slice следующей недели.
- Если остались силы и контекст стабилен:
  - исправить seed exit semantics;
  - добавить targeted test на ненулевой exit при ошибке.
- Записать:
  - что завершено;
  - что подтверждено экспериментом;
  - что перенесено;
  - какие проверки не запускались.

**Достаточный результат дня:** следующая неделя начинается с конкретной реализации, а не с повторного исследования.

## Порядок задач при нехватке сил

1. Сохранить здоровье и возможность продолжать работу.
2. Закрыть текущий PR/release-контекст.
3. Принять и записать стратегию PostgreSQL.
4. Провести эксперимент с чистой PostgreSQL.
5. Исправить seed semantics.

Всё ниже пропущенного пункта переносится без попытки наверстать ночью или в выходные.

## Короткий ежедневный журнал

В конце рабочего блока достаточно трёх строк:

```text
Сделано:
Обнаружено:
Следующий шаг:
```

Отдельно можно записать состояние по шкале 1-5. При значении 1-2 два дня подряд план уменьшается, а не уплотняется.
