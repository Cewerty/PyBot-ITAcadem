# План развития PyBot ITAcadem на май 2026

## Цель месяца

До 31 мая 2026 привести проект к более безопасному и сопровождаемому MVP-состоянию для возможной эксплуатации клуба: закрыть базовые риски, зафиксировать архитектурные решения, добавить минимальные админские возможности и не начинать крупные платформенные эпики.

## Eisenhower ABCD

Код:

- `A` - важно и срочно: делать в первую очередь.
- `B` - важно, но не срочно: планировать после Must-задач.
- `C` - срочно или удобно, но не стратегически критично: делать только при наличии ресурса.
- `D` - не брать в майский scope.

## MoSCoW + SMART

### Must Have

#### 1. Провести код-ревью проекта (✅ Выполнено)

- `Specific`: пройти ключевые слои: handlers, services, repositories, DI, notification, broadcast, role request.
- `Measurable`: оформить список найденных проблем в GitHub issue/checklist с приоритетами.
- `Achievable`: без переписывания проекта и без крупных refactor-эпиков.
- `Relevant`: снижает риск перед эксплуатацией клуба.
- `Time-bound`: до 31 мая 2026.

#### 2. Провести базовый security-аудит (✅ Выполнено)

- `Specific`: проверить логи с персональными данными, роли/доступы, токены/env, registration contact self-check и admin-only команды.
- `Measurable`: оформить список рисков и исправить минимум 2-3 риска высокого приоритета.
- `Achievable`: без внедрения полной security-платформы.
- `Relevant`: снижает риск утечек и ошибочного доступа.
- `Time-bound`: до 31 мая 2026.

#### 3. Добавить ADR о системе адаптеров (✅ Выполнено)

- `Specific`: зафиксировать ports/adapters, расположение портов и адаптеров, отношение к Telegram, TaskIQ и notification.
- `Measurable`: добавить новый ADR в `src/pybot/docs/adr/`.
- `Achievable`: документ без обязательной большой миграции кода.
- `Relevant`: снижает архитектурную неопределенность вокруг внешних интеграций.
- `Time-bound`: до 31 мая 2026.

#### 4. Добавить команду просмотра пользователей

- `Specific`: добавить admin-only команду, например `/users` или `/users_by_role`, с лимитом/пагинацией.
- `Measurable`: команда покрыта bot/service тестами на happy и unhappy paths.
- `Achievable`: использовать существующие repositories/services и role middleware.
- `Relevant`: помогает администрировать клуб без прямого доступа к БД.
- `Time-bound`: до 31 мая 2026.

### Should Have

#### 5. Добавить ADR об изменении структуры папок (✅ Выполнено)

- `Specific`: описать будущую структуру для adapters, notification, task flows и texts/renderers.
- `Measurable`: добавить ADR без обязательного массового переноса файлов.
- `Achievable`: документировать решение и последствия.
- `Relevant`: снижает риск хаотичных перемещений при следующих фичах.
- `Time-bound`: до 31 мая 2026, если Must-задачи не съели месяц.

#### 6. Минимально адаптировать рассылку под фоновый формат

- `Specific`: добавить минимальный background-flow через TaskIQ.
- `Measurable`: есть `BroadcastRun` или эквивалентный статус выполнения, TaskIQ task и запись результата доставки хотя бы агрегировано.
- `Achievable`: без полноценного EIP, outbox и distributed delivery guarantees.
- `Relevant`: снижает риск долгих Telegram handlers и частичных рассылок.
- `Time-bound`: до 31 мая 2026 при наличии ресурса.

#### 7. Настроить лидерборд в чат/топик супергруппы по расписанию (✅ Выполнено)

- `Specific`: добавить настройки `chat_id`, `message_thread_id` и интервала/расписания для публикации лидерборда.
- `Measurable`: есть тесты конфигурации и вызова weekly publisher.
- `Achievable`: использовать существующий TaskIQ-контур.
- `Relevant`: поддерживает клубную мотивацию без ручной публикации.
- `Time-bound`: до 31 мая 2026 при наличии ресурса.

#### 8. Добавить поддержку Telegram member tags

- `Specific`: добавить интеграцию с Telegram Bot API для установки member tags в супергруппе.
- `Measurable`: есть порт/адаптер или сервис синхронизации, настройки супергруппы и тесты маппинга ролей/статусов в тег.
- `Achievable`: минимальный scope без полноценной системы тегов внутри приложения.
- `Relevant`: помогает визуально отображать роли/статусы участников клуба в Telegram.
- `Time-bound`: до 31 мая 2026 при наличии ресурса и прав `can_manage_tags` у бота.

### Could Have

#### 9. Перевести 1-2 сложные команды на staged-flow

- `Specific`: выбрать самые болезненные команды, например начисление баллов или управление компетенциями.
- `Measurable`: переведены не более 1-2 команд, добавлены flow tests.
- `Achievable`: без переписывания всех handlers.
- `Relevant`: снижает сложность парсинга и UX-ошибки.
- `Time-bound`: только после Must и Should задач.

#### 10. Описать минимальный audit/business operation ledger

- `Specific`: описать события аудита: actor, target, operation, reason, metadata, timestamp.
- `Measurable`: есть design note/ADR или issue с моделью данных и границами сервиса.
- `Achievable`: без полной реализации audit-log платформы.
- `Relevant`: готовит основу для разборов спорных операций.
- `Time-bound`: только если останется ресурс.

#### 11. Сделать SQLAdmin proof-of-concept

- `Specific`: локально проверить подключение SQLAdmin к ключевым бизнес-сущностям.
- `Measurable`: есть PoC branch/notes, список рисков и вывод go/no-go.
- `Achievable`: без production-доступа и без JWT/API.
- `Relevant`: может упростить администрирование, но повышает security risk.
- `Time-bound`: только если останется ресурс.

### Won't Have в мае 2026

Не брать в майский scope:

- полноценную систему задач;
- PostgreSQL migration;
- REST API;
- JWT auth/authn;
- Grafana + Alloy + Loki;
- FastStreams/EDA;
- multi-instance support;
- интеграцию API колледжа с Circuit Breaker;
- полноценную audit-log платформу.

## Критерий успешного месяца

Май считается успешным, если до 31 мая 2026 выполнены:

- код-ревью;
- базовый security-аудит;
- ADR о системе адаптеров;
- команда просмотра пользователей;
- минимум 2-3 точечных hardening-fix по результатам аудита.

Фоновая рассылка, scheduled leaderboard и Telegram member tags считаются бонусом, а не обязательством.
