# ADR 018: Контракт пользовательского контекста для публичных info-команд

## Дата: 20-06-2026

## Статус: Accepted

## Контекст

Некоторые публичные Telegram-команды могут работать в двух режимах:

1. просмотр данных явно указанного пользователя через reply, text mention или Telegram ID;
2. просмотр данных текущего пользователя, если он уже зарегистрирован в системе.

После добавления `UserActivityMiddleware` доменный `user_id` текущего отправителя стал попадать в handler data только для пользователей, уже существующих в БД. Если такая команда требует обязательный `user_id: int` в сигнатуре handler-а, незарегистрированный отправитель не может вызвать даже режим explicit target, хотя для него текущий доменный пользователь не нужен.

## Решение

Для публичных info-команд, которые умеют работать с explicit target, `user_id` текущего отправителя должен быть необязательным: `int | None = None`.

Handler обязан различать три ситуации:

1. explicit target найден -> команда работает с доменным `target_user.id`;
2. explicit target отсутствует, `user_id` есть -> команда использует self-fallback через доменный `user_id`;
3. explicit target отсутствует, `user_id` отсутствует -> команда отвечает понятным presentation-text сообщением о необходимости регистрации для просмотра своих данных.

Telegram ID текущего отправителя нельзя использовать как замену доменному `user_id` для self-fallback. Telegram ID остаётся транспортным идентификатором и может использоваться только для разрешения явно указанной цели в пользователя системы.

Если explicit target указан, но не найден, команда должна продолжать отвечать ошибкой target-not-found, а не сообщением о регистрации текущего отправителя.

## Последствия

### Положительные

- [+] Незарегистрированные пользователи могут использовать публичные info-команды в режиме просмотра явно указанной цели.
- [+] Self-lookup остаётся привязан к доменному пользователю из БД.
- [+] Будущие info-команды получают явный контракт и не повторяют неявный registration-gate через обязательный `user_id`.

### Отрицательные

- [-] Handler-контракт становится чуть шире: вызывающий код и тесты должны учитывать `user_id is None`.
- [-] Для команд нужны отдельные пользовательские тексты на ветку no target + no user context.
